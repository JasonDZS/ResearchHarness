# ResearchHarness v0.1 Implementation Plan

Date: 2026-04-10
Status: Draft
Repository: `JasonDZS/ResearchHarness`

## Goal

Ship `v0.1` as a single-session interactive research harness that proves the core product loop:

- start a research session from a goal
- maintain explicit tasks and active focus
- search literature and gather evidence inside the same session
- write durable artifacts during the session
- preserve provenance for writing outputs
- pause, resume, and redirect work without losing context

This version should prove product identity, not maximize surface area or autonomy.

## Non-Goals

The following are out of scope for `v0.1`:

- background agent swarms
- MCP integration
- plugin marketplace
- remote research workers
- autonomous unattended execution
- browser or desktop UI
- multi-user collaboration
- environment-level reproducibility capture

## Release Criteria

`v0.1` is complete only when all of the following are true:

1. A user can start, pause, and resume the same research session.
2. A user can inspect and steer explicit tasks during the session.
3. Literature tools can gather and persist a nontrivial source set.
4. Artifacts are written incrementally by workstream.
5. Draft outputs include citation provenance and section source mapping.
6. The user can redirect work mid-session without restarting the session.

Execution guidance:

### Issue 1 — P0: Project Skeleton and Core Domain Models

Why first:

- everything else depends on stable package boundaries and typed persistence shapes

Depends on:

- none

Requirements:

- create the Python package skeleton for shell, session, tools, persistence, artifacts, llm, and domain
- define typed models for `ResearchSession`, `Task`, `Checkpoint`, `ArtifactRef`, and provenance records
- create `.research/` workspace layout helpers
- add configuration loading and environment validation
- add a minimal `rh` CLI entrypoint

How to test:

- create unit tests for model validation and serialization
- create a smoke test that initializes a workspace and creates session directories
- run CLI smoke test to verify `rh` starts successfully

Acceptance criteria:

- package structure exists and imports cleanly
- session/domain models round-trip between memory and disk
- `.research/session/` and `.research/artifacts/` can be created from a clean repo
- `rh` can be invoked without crashing

### Issue 2 — P0: Session Store, Pause/Resume, and Recovery

Why second:

- session continuity is the core product promise

Depends on:

- Issue 1

Requirements:

- implement `SessionStore` for session metadata and transcript persistence
- implement `ResumeManager` and recovery loading paths
- support `rh`, `rh resume`, `rh status`, and `rh pause`
- persist active focus, current plan snapshot, and task pointer
- handle interrupted sessions safely

How to test:

- add unit tests for session persistence and reload
- add integration tests for start -> pause -> resume flow
- add interrupted-session recovery test that verifies state is surfaced on resume

Acceptance criteria:

- a session can be resumed without restating the research goal
- transcript, active focus, and metadata survive restart
- interrupted sessions do not corrupt persisted state
- `rh pause` exits only after a safe boundary is persisted

### Issue 3 — P0: Shell Commands, Status View, and Input Normalization

Why now:

- the harness must be steerable and legible before it becomes more capable

Depends on:

- Issue 1
- preferably Issue 2 for full `resume` and status integration

Requirements:

- implement shell command registry and input normalization
- support `/help`, `/status`, `/plan`, `/tasks`, and `/checkpoint`
- render current session status, active task, workstream, and pending approvals
- distinguish natural-language requests from explicit shell commands

How to test:

- add parser tests for command routing and normalization
- add snapshot or string-output tests for command rendering
- run interactive smoke tests for each built-in command

Acceptance criteria:

- built-in commands resolve consistently
- status output shows active task, workstream, and session state
- command handling does not break natural-language turn flow
- user can inspect the runtime without leaving the session

### Issue 4 — P0: Task Store, Planner, and Task Mutation Flow

Why now:

- explicit tasks are required for legibility and user steering

Depends on:

- Issue 1
- preferably Issue 2 so task state and session continuity evolve together

Requirements:

- implement `TaskStore` persistence
- implement `TaskPlanner` for creation, reprioritization, and focus selection
- support task statuses: `pending`, `in_progress`, `blocked`, `completed`, `cancelled`
- support `/task add <text>` and `/task focus <id>`
- add internal task mutation tools for runtime updates
- link tasks to workstreams and artifact references

How to test:

- add unit tests for task CRUD and status transitions
- add integration tests for user-driven task add/focus flows
- test runtime-driven task updates and persistence after resume

Acceptance criteria:

- users can inspect, add, and focus tasks
- the runtime can update tasks without corrupting task state
- tasks survive pause/resume
- tasks visibly track workstream and artifact references

### Issue 5 — P1: Research Query Loop and Literature Tool Runtime

Why next:

- this is where the harness starts doing real research work

Depends on:

- Issue 1
- Issue 2
- Issue 3
- Issue 4

Requirements:

- implement `ResearchSessionRuntime` orchestration around the query loop
- implement `PromptContextBuilder`

How to test:

Acceptance criteria:

- the runtime can execute a tool-assisted turn in one continuous session
- literature tool results are normalized and reusable
- tool results feed back into the active turn state
- file writes remain scoped to artifact roots

### Issue 6 — P1: Artifact Store and Workstream Output Contracts

Why next:

- `v0.1` must produce durable research outputs, not only chat logs

Depends on:

- Issue 1
- ideally Issue 5 before final integration into the runtime loop

Requirements:

- implement `ArtifactStore`
- organize artifacts by workstream
- support incremental writes for literature, analysis, and writing artifacts
- create typed artifact serializers and manifest/index metadata
- expose artifact refs back into the session context

How to test:

- add unit tests for artifact path resolution and serialization
- add integration tests for incremental artifact writes across workstreams
- verify artifact refs appear in session state after writes

Acceptance criteria:

- artifacts are written into workstream-specific directories
- repeated writes are handled safely
- artifact refs are tracked in session/task state
- durable outputs exist independent of transcript history

### Issue 7 — P1: Provenance, Writing Outputs, and Checkpoint/Approval Flow

Why next:

- traceable writing is the key differentiator from generic copilots

Depends on:

- Issue 5
- Issue 6

Requirements:

- implement `ProvenanceStore`
- support writing outputs:
  - `paper_draft.md`
  - `citation_provenance.json`
  - `section_source_map.json`
- implement `CheckpointManager`
- define approval rules for:
  - major artifact overwrite
  - replacing the active plan
  - abandoning/superseding a task
  - final writing export
- support manual `/checkpoint`

How to test:

- add unit tests for provenance record generation
- integration test draft generation with required provenance sidecars
- integration test approval-required flows for risky semantic actions
- test checkpoint summaries after major artifact creation

Acceptance criteria:

- writing artifacts remain source-traceable
- citation provenance exists for generated draft content
- checkpoint summaries can be created manually and automatically
- risky semantic actions pause for explicit user approval

### Issue 8 — P2: Session Memory, Compaction, and v0.1 Hardening

Why last:

- this improves durability and real-world usability after the core loop already works

Depends on:

- Issue 2
- Issue 4
- Issue 5
- Issue 6
- Issue 7

Requirements:

- implement typed durable memory store with categories:
  - `user`
  - `feedback`
  - `project`
  - `reference`
  - optional `research_direction`
- implement session compaction for long transcripts
- preserve active focus, pending approvals, key findings, and artifact refs during compaction
- improve error handling and recovery robustness for end-to-end flows

How to test:

- add unit tests for memory categorization and persistence
- integration test long-session compaction without losing active task continuity
- recovery test for interrupted turn after compaction
- final end-to-end QA on the full `v0.1` loop

Acceptance criteria:

- memory remains separate from artifacts and transcript truth
- compaction preserves enough context for seamless continuation
- resumed sessions recover correctly after interrupted active work
- the full `v0.1` release criteria can be demonstrated in a realistic flow

## Implementation Notes

- `v0.1` should remain single-session and sequential
- legibility is more important than premature autonomy
- artifacts are the durable source of research truth
- provenance must be treated as required product behavior, not optional polish
- memory should support continuity, not replace evidence

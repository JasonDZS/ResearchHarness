# ResearchHarness v0.1 Implementation Plan

Date: 2026-04-10
Status: Draft
Repository: `JasonDZS/ResearchHarness`

## Goal

Ship `v0.1` as a single-session interactive research harness that proves the core product loop:

- start a research session from a goal
- maintain explicit tasks and active focus
- search literature and gather evidence inside the same session
- write durable artifacts during the session
- preserve provenance for writing outputs
- pause, resume, and redirect work without losing context

This version should prove product identity, not maximize surface area or autonomy.

## Non-Goals

The following are out of scope for `v0.1`:

- background agent swarms
- MCP integration
- plugin marketplace
- remote research workers
- autonomous unattended execution
- browser or desktop UI
- multi-user collaboration
- environment-level reproducibility capture

## Release Criteria

`v0.1` is complete only when all of the following are true:

1. A user can start, pause, and resume the same research session.
2. A user can inspect and steer explicit tasks during the session.
3. Literature tools can gather and persist a nontrivial source set.
4. Artifacts are written incrementally by workstream.
5. Draft outputs include citation provenance and section source mapping.
6. The user can redirect work mid-session without restarting the session.

## Release Validation

In addition to the high-level release criteria above, `v0.1` should not be signed off until all of the following validation checks are demonstrated:

- At least one end-to-end research flow from literature search to partial draft works in one continuous session.
- Verified literature sources in a realistic session reach the PRD threshold.
- Provenance coverage is complete for generated draft citations.


Execution guidance:

- `Plan Item 1` must land first.
- `Plan Item 2` should land next because session continuity is the core product promise.
- `Plan Item 3` should land before the persistent shell work because it provides the shared command layer and normalization logic.
- `Plan Item 4` should land next to close the current gap between command support and a true session-first terminal UX.
- `Plan Item 5` should start after session, command, and interactive shell foundations are usable.
- `Plan Item 6` can begin in parallel with the later part of `Plan Item 5`, but its final integration depends on the runtime/tool loop.
- `Plan Item 7` depends on both the runtime loop and the artifact layer.
- `Plan Item 9` is intentionally last because it hardens the complete system rather than defining the base loop.

### Plan Item 1 — P0: Project Skeleton and Core Domain Models

GitHub tracker: [#2](https://github.com/JasonDZS/ResearchHarness/issues/2)

Completion status: Completed in `main`

Why first:

- everything else depends on stable package boundaries and typed persistence shapes

Depends on:

- none

Requirements:

- create the Python package skeleton for shell, session, tools, persistence, artifacts, llm, and domain
- define typed models for `ResearchSession`, `Task`, `Checkpoint`, `ArtifactRef`, and provenance records
- create `.research/` workspace layout helpers
- add configuration loading and environment validation
- add a minimal `rh` CLI entrypoint

How to test:

- create unit tests for model validation and serialization
- create a smoke test that initializes a workspace and creates session directories
- run CLI smoke test to verify `rh` starts successfully

Acceptance criteria:

- package structure exists and imports cleanly
- session/domain models round-trip between memory and disk
- `.research/session/` and `.research/artifacts/` can be created from a clean repo
- `rh` can be invoked without crashing

### Plan Item 2 — P0: Session Store, Pause/Resume, and Recovery

GitHub tracker: [#3](https://github.com/JasonDZS/ResearchHarness/issues/3)

Completion status: Completed in `main`

Why second:

- Plan Item 1

- implement `SessionStore` for session metadata and transcript persistence
- implement `ResumeManager` and recovery loading paths
- support `rh`, `rh resume`, `rh status`, and `rh pause`

How to test:

- add unit tests for session persistence and reload

Acceptance criteria:

- a session can be resumed without restating the research goal
- transcript, active focus, and metadata survive restart
- interrupted sessions do not corrupt persisted state
- `rh pause` exits only after a safe boundary is persisted

### Plan Item 4 — P0: Terminal Interactive Shell / REPL

GitHub tracker: [#15](https://github.com/JasonDZS/ResearchHarness/issues/15)

Completion status: Not started

Why now:

- this closes the current gap between the shell command layer and the session-first UX promised in the roadmap

Depends on:

- Plan Item 2
- Plan Item 3

Requirements:

- implement `ResearchShell` as a persistent terminal loop
- make `rh` enter the interactive shell by default when no one-shot subcommand is requested
- reuse the existing shell command registry and input normalization inside the interactive loop
- support mixed natural-language turns and slash commands in one continuous process
- support safe exit paths for interactive use, including `/pause` and `/exit`
- surface session status and recovery context inside the shell so the user can steer without restarting the process

How to test:

- integration tests for multi-turn interactive session flow in one process
- smoke tests for slash commands and natural-language turns inside the same shell session
- pause/exit tests that verify state is persisted before the shell stops

Acceptance criteria:

- running `rh` starts a persistent terminal session loop instead of a one-shot wrapper
- the user can send multiple natural-language requests and slash commands without restarting the process
- session state survives pause/exit and can be resumed later
- interactive command handling remains consistent with the non-interactive command layer

## Implementation Notes

- `v0.1` should remain single-session and sequential
- legibility is more important than premature autonomy
- artifacts are the durable source of research truth
- provenance must be treated as required product behavior, not optional polish
- memory should support continuity, not replace evidence
